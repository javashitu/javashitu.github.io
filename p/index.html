分词器

一般在语料中存在身份证，电话号码等信息都需要脱敏，因为这些语料是高基数且不重复的容易产生大量的低价值token片段。这会影响模型学习语言的规律。但是如果需要模型学习里识别里面的特定实体，那么则相反，这些语料就需要保留。还有就是在多语言或者混合语料中，应该考虑要准备对低资源语料做采样或者保留，否则可能因为高频语料主导模型导致模型的语言性能。

预分词是指将原始文本切分成可以统计，合并的基础单元，比如字符，字节或者Unicod片段，常见的分词手段有按照空格，自字符，Unicode类别，甚至字节划分。

  • 按照标间符号，空格拆分
  • 按照unicode类别拆分，比如按照数字，标点，中文，特殊字符拆分token，这样不同的unicode会归属到不同的token块。这种方法适合多语言混合的文本。
  • 字节级拆分，完全不依赖语言，字符，种类，按照字节序列得到单个token

前两种策略在处理长缺乏显示分隔符合长段同类型文本时，可能会退化成字符级切分，但是第三种不会。因此UTF-8策略具备最强通用性，能减少未登录词(OOV)的出现。对于以空格为词边界的语言，可以按照正则等边界拆分，但是对于中文，则一般按照字符拆分。预分词最大的好处就是提升TOKEN利用率，减少未登录词的情况。

预分词的目的是为了实现高效的模型输入，因为基于词组的原始的词表可能有几万个，但是基于字节的词表只有265个，这样输入的维度就变得更小，更方便模型预测接一个字符后的字符，比如e5a5bd 表示好，模型发现这三个字节经常在一起出现，那么当看到e5时就会猜测后面的是不是a5,bd

上面讲的还是词的预处理，主要目的是构建原始数据，其中t是组成词表的元素，而tokens则是词的数据表征。虽然这里得到了词/句的数据表示，但是不会这么直接的就把数组直接丢给模型，因为模型的计算是合数据的维度正相关的，比如一本书有几千万字节的词，把这个几千万个字节长度的输入进模型，复杂度直接N^2^爆炸。而且，单个字节，比如E6携带的有效信息太少，模型要在这265个词表里学习出来特征，会变得很复杂(直接原始自然语言词表也包含特征，但是此表太大，几万个)。算法的目的就是为了在模型输入合原始数据之间找到平衡点，讲这些字符级词表，按照一些规则组装成比较适合模型的输入格式。

原始句子水石头，第一步的预处理是把石头打成沙子，算法就是把沙子做成砖头。

有了词表合词的表征，接下来就要靠算吧把他们的特征计算出来，算法有

  • BPE
  • WordPiece
  • Unigram
  • SentencePiece

这些算法各自有自己的特点，但是他们的目的都是为了让划分出来的token更加清晰一致（最大化语料似然），从而间接提升模型的性能。这些算法的最终目标是生成词表（比如要求达到32K或者50K大小的词表），一般衡量这些算法的优劣有平均token长度的里的未登录词率（oov），压缩率(总字符数/总token数)

常见分词器

  1. 字符分词器 将文本拆分成最小的字符单元，词表非常小，英文只需要26个字母，中文只需要几千个常用字。也没有OOV问题，但是序列会非常长，非常消耗LLM的上下文。而且单个字符比如T一般不具备语义，模型很难理解，所以需要加深网络层数。
  2. 字节分词器 任何语言的底层存储都是字节，那么所有的字都可以转化为256大小的基础词表，一般现代模型,比如LLMA不会纯使用字节分词器，而是将字节作为BPE的基础单元即BBPE。注意字节分词器的压缩率恒等于1，因为输入文本的字节数量=总token数量
  3. 词级分词器 深度学习早期时代，主流方法，基于空格或者分词算法，将文本切分成具备独立意义的词。他的问题是词表太大(几十万，上百万)OOV严重，任何没登录的人名，地标建筑都识别不了
  4. BPE分词 现代LLM的主流分词方法，目的是为了在字符合词级之间找到平衡，一般过程是
    • 初始化，将单次拆分成字符序列
    • 统计所有相邻字符出现的频率(比如e和s经常出现)
    • 将出现频率最高的合并成新的token,比如es
    • 循环上述步骤，直到达到预设的词表大小

加载deepseek的分词器，实际加载后，可能会出现基于token生成的字符串与原始的文本有差异，这是因为LLM所使用的词表在预训练过程中对某些字符或者子词的覆盖不足，导致模型无法生成对应的token，在还原时自然就丢失了这部分内容导致看起来可能是乱码，解决方式是增加训练语料或者进行更加充分的BPE训练，建立更加完整的token词映射表。

思考

  1. 研究表明，视觉特征能增强LLM的表达能力，但并不是适用于所有的任务，那么是否可以在视觉表征与离散token之间寻求平衡点，同时为模型提供两种表征方式，并且借鉴MOE的思想设计轻量级路由，使模型能在不同的任务或者语言之间自动选择或者融合最合适的词，从而提升模型的性能。
  2. 能否设计一种“自适应分词器”，并且让分词器和llm分开训练，并且通过特殊机制让分词器与模型结合起来，使得模型在执行任务时，分词器仍然能够动态学习和优化token的划分策略。
  3. 能否借助微分子词模块，元学习和强化学习等方法，让分词器能从少量对话或者任务样本中自动发现最合适的token划分方式，从而降低模型的任务对数据的依赖性并且提升模型的泛化能力。

时间估算

FLOPS（floating point operations）浮点运算次数，表示一个模型在执行过程所需要的执行的浮点加减乘除的运算总次数。经验公式是

总计算量 ≈ 6 * 参数车辆 * token数量

6倍是因为前向传播forward pass需要计算加法 + 乘法的计算，就是2倍，反向传播backword pass需要计算梯度，约等于前向传播的2倍工作量

比如训练一个70B参数的模型，使用15T数据，那么就需要

6 * 70 * 10^9^ * 15 * 10^12^ ≈ 6.3 * 10 ^24^ FLOPS 的计算量

如果使用H100显卡(1979 TFLOPS)计算，巅峰算力1979 TFLOPS，万亿级别，但是这个算力是使用了FP16或者BF16 ,并且开启结构化稀疏(structured sparsity)才能达到。一般的普通稠密模型(llm3,qwen3等)，理论性能减半约990 TFLOPS

结构化稀疏是指的一种模型压缩方法，将稠密模型按照50%稀疏度剪枝，称为n:m，也就是连续m个权重必须减掉n个比如2:4,4:8。非结构化剪枝性能最差因为需要随机

显卡的理论性能(MFU model flops utilization 模型利用率)一般跑不满，一般是30%-60%,计算时可以取50%

那么假设赢1024张H100来训练这个模型，就需要时间

6.3 * 10^24^ / (5 * 10^17^) = 1.26 * 10^7^ 秒 = 146天

张量

模型的参数，梯度，优化器状态，输入数据，等等数据都是以张量存储的。大多数对张量的操作都是从其他的张量来的。许多的操作只会提供张量的视图view不会创建新的张量。因此修改一个张量也会影响另一个。

张量的view在内存中可能不是按顺序存储那么就是不连续的，不能对非连续的张量再次执行某些操作比如view,如果需要，需要先对张量执行.contiguous()方法，它会将张量复制到新的内存中。

可以对张量进行逐元素操作(elements wise operations)，也就是对每个元素分别进行某种操作，比如x.pow(2) 或者x+x，这种运算就是逐元素操作。矩阵乘法使用 @ 连接，矩阵的乘法时会自动执行只计算后面一个tensor的最后两个维度

因果注意力掩码(causal attention mask)函数triu，也就是大模型里的上三角矩阵，保证第j个词只能看到第j个词之前的词。这种上三角矩阵m[i,j]表示i位置对于j位置的贡献，当i比j大时(i在j之后，贡献应该为0),默认3 x 3的全1张量x，x.triu()输出

内存

  • FP32 一般用于炼丹时存储参数的主副本和优化器状态
  • FP16 上一代混合精度训练时使用，基本被BF16取代
  • BF16 神经网络对小数点后的几位不敏感，但是对数值范围敏感，所以设计了8位指数位+7为尾数位，当前训练的主流选择
  • FP8 以及他的各种变体E5M2，推理阶段量化

numpy和pytorch的张量在底层都是行存储，每个tensor内部包含步长strides和指针，指针指向实际的内存，比如二维tensor里，步长就表示每一行的元素数量，要移动到下一行就只需要跳过指定步长的元素。view操作不会复制底层内存，所以很多操作不支持

FLOPs表示浮点运算的总次数，FLOPS(FLOP/s)表示每秒浮点运算次数

以线性模型为例，计算次数等于 2 * 数据量 * 参数数量，transformer也适用这个公式，因为大部分计算来自矩阵乘法(90%)

MFU(Model FLOPs Utilization)模型性能利用率 = 实际FLOPS/峰值FLOPS，一般大于0.5视为还不错的性能，达到1基本不可能

关于梯度的计算，本质上x和y共同经过计算组成一组计算图，直到loss，当执行反向传播时，则是以loss为根，计算到x和y的梯度，x和y是这个计算图的叶子结点，也是真正需要计算出梯度的节点。中间的节点产生的梯度pytorch会删除。但是在pytorch里不是所有的参与计算的tensor都是叶子结点，叶子结点必须是手动创建的，不能是中间计算过程产生的(pred_y就不是)，而且必须指定required_grad = True，否则会被当做常量。

    @classmethod
    def backward_compute(self):
        x = torch.tensor([1., 2, 3])
        y = torch.tensor([1., 1, 1], requires_grad = True)
        # 一维时是计算向量点积
        pred_y = x @ y
        print(pred_y)
        loss = 0.5 * (pred_y - 5).pow(2)
        # 触发反向传播
        loss.backward()
        # loss是计算图的根，梯度等于1，他的梯度在计算过程里删除了
        assert loss.grad is None
        # 中间变量的梯度会在计算后删除，所以是none
        assert pred_y.grad is None
        # 保留grad
        pred_y.retain_grad()
        assert x.grad is None
        # y的梯度就是x的值
        assert torch.equal(y.grad, torch.tensor([1,2,3]))

自定义模型

Torch.random是标准高斯分布初始化，这种初始化随着层数变深，数值会非常大，所以需要优化，一般是通过除以输入维度的平方根来解决。除此之外还需要o将随机数控制在一个合理的范围内(xavier初始化)，避免高斯分布的尾部产生极端值。

在dim = 3,layer = 4的场景下。我在init方法里用moduleList创建出来了4个layer，也就是说这些layer就是相连接的，最后又手动定义了一个layer final,这个final可以认为时输出层。在实际的计算时，x是输入，那么B,D就是x的输入维度，D应该是等于3，B则是这一批样本的数量。在计算时就是不断的把样本x输入进每一层layer，每层layer里会执行计算。最后我再把4层layer的计算结果调用final手动计算。最后的squeeze(-1)是因为我的计算结果是一个B行1列的矩阵，squeeze会把矩阵压缩成张量，最后得到的就是一个长度B的张量

class Cruncher(nn.Module):
    # 以dim = 3， layer = 4为例
    def __init__(self, dim: int, num_layer: int):
        super().__init__()
        self.dim = dim
        # 这里其实是创建了4个layer放进了ModuleList里，layer之间并没有形成全连接
        self.layers = nn.ModuleList([
            Linear(dim, dim)
            for i in range(num_layer)
        ])
        self.final = Linear(dim, 1)

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        # x表示一批数据，数据里有B个样本，每个样本有D个维度
        B,D = x.size()
        # 因为有这个计算，4层layer之间才形成了全连接
        for layer in self.layers:
            x = layer(x)
        # 最后使用final层进行计算，得到最终的输出结果，因为有B个样本，最终计算得出的结果其实就是B个维度为1的tensor
        # 这里注意，这里的B个维度为1的tensor不是[1,2,3]这种格式而是[[1],[2],[3]]这种格式，所以需要最后的squeeze
        x = self.final(x)
        assert x.size() == torch.Size([B, 1])
        # squeeze几挤压掉最后一个维度，会把tensor编程[1,2,3这种格式]
        x = x.squeeze(-1)
        assert x.size() == torch.Size([B])
        return x

class ModelTrain:
    def __init__(self):
        self.dataPath = "../../../data/chapter3/data.py"
        origin_data = np.array([1, 2, 3, 4, 5, 6, 7, 8], dtype=np.int32)
        origin_data.tofile(self.dataPath)

    @classmethod
    def get_batch(self, data: np.ndarray, batch_size: int, sequence_length: int, device: torch.device) -> torch.Tensor:
        # 1. 随机选择 B 个起始位置
        # 这里的上限是 len(data) - sequence_length，防止切片越界
        start_indices = torch.randint(0, len(data) - sequence_length, (batch_size,))

        # 2. 根据索引提取数据并组成矩阵 (B, L)
        # 这里的逻辑是：对每一个 start 索引，取 data[start : start + sequence_length]
        x = torch.stack([torch.from_numpy(data[i:i + sequence_length].astype(np.int64)) for i in start_indices])

        # 3. 把数据移动到对应的设备上（CPU/GPU/MPS）
        x = x.to(device, non_blocking = True)

        # 4. 固定内存优化（仅限 CUDA）
        if device.type == 'cuda':
            x = x.pin_memory()

        return x

modelTrain = ModelTrain()
data = np.memmap(modelTrain.dataPath, dtype=np.int32)
B = 2
# 虽然这里叫做L，但是这个值其实对应在模型里的D，这两个值必须一致，否则运算会失败
# 这里之所以叫做L是因为在Transformer模型里这个设置一般是词元的数量，也就是一句话里词的数量
L = 4
device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
x = ModelTrain.get_batch(data, B, L, device=device)
print(x)
assert x.size() == torch.Size([B,L])

让我来捋一下这个代码的逻辑，代码先定义了一堆参数并用这些参数定义了模型和优化器还有损失函数。然后开启循环，每次循环里取出的一批数据有32条，并且每条数据4个特征维度。然后随机初始化一个包含32个元素的向量来作为目标值y_true。再开始计算之前调用zere_grad清空梯度，为什么需要这一步我猜测是因为optimizer会在每一批数据之间累计梯度，这些梯度指的就是model里每一层layer里的参数的梯度。如果不手动清空就会累计上上一批次的梯度。之所以梯度会累计，是因为设计如此，因为如果在比较小的显存GPU上执行，一次需要输入很大的批次数据，但是没办法全量输入，就可以分解为多个小批次的数据，这样最后累计出来的梯度和一次输入整个批次还是一样的。

然后调用损失函数criterion计算真实值和预测值的差距。然后计算损失值loss关于layer里各个参数的梯度，也就是执行backward。然后根据损失值的大小，由优化器optimizer来调整模型里各个layer的参数。

为什么loss能执行反向更新，loss是声明出来的损失函数的计算值，它确实跟模型的输出结果y_pred有关系，但是我没看到它怎么和模型model本身有什么绑定关系，那么他是怎么绑定上模型的各个层，并且执行的反向传播呢。实际上loss并不是一个值，loss是关于y_pred的计算图，y_pred本身是由final层计算得出的，loss则是在final后接上的心得计算，这就形成了从final到前面所有层参数的计算图，也正是有这个图，所以loss能顺着final找到前面每一层得参数，然后为每个参数更欣赏梯度，每个参数其实有一个.grad 属性，里面装的就是参数的梯度。backward时就是去填充这些grad

optimizer也是类似的问题，optimizer本身并没有和model有关联，它又是怎么能更新上model的参数，以及它怎么知道loss的值，并且根据loss来判断应该怎么更新的。其实oprimizer在声明时就和model的所有parameter关联上了，这样optimizer就能找到所有参数的位置，也能知道参数的梯度，这样才能更新参数。还有一个点就是loss函数是被设计成连续可导并且loss的计算值是始终大于等于0的，所以在实际更新时optimizer为了让loss值降低，需要做的就是沿着导数相反的方向去更新参数就能达到降低loss值的目的，比如导数为正数，说明增加参数会导致loss值增大，所以减小参数。如果导数为负数说明减小参数会导致loss值增加，所以增大参数。这样就把原本关于loss值大小和参数的关系转换到了loss导数和参数的关系。

还有就是loss函数虽然一般都是二元一次函数，但是因为llm是在多层之间互相计算的，这导致最终在loss里组合出来的计算实际是一个高次多项式方程，这也是为什么loss函数的图像会呈现山脉的效果

from llm.diy.chapter3.cruncher import Cruncher
from torch.optim import SGD
import torch.nn.functional as F
import torch

from llm.diy.chapter3.model_train import ModelTrain


class LoopTrain:
    checkpoint_path = "../../../data/chapter3/model_checkpoint.pt"

    def train(self, name: str, get_batch, D: int, num_layers: int, B: int, num_train_steps: int, lr: float):
        # 当num_layers = 0时，就只有final一层，就是简单的线性回归
        model = Cruncher(dim=D, num_layer=num_layers)
        # 创建优化器
        optimizer = SGD(model.parameters(), lr=lr)
        if "simple2" == name:
            load_checkpoint = torch.load(self.checkpoint_path)
            model.load_state_dict(load_checkpoint["model"])
            optimizer.load_state_dict(load_checkpoint["optimizer"])

        for t in range(num_train_steps):
            # 加载输入数据，因为是传入了一个创建数据的方法，所以这里创建出来的数据每次都不一样
            x, y = get_batch(B=B)
            # 向前传播并且计算损失
            pre_y = model(x)
            loss = F.mse_loss(pre_y, y)
            # 反向传播计算梯度
            loss.backward()
            # 修改参数
            optimizer.step()
            # 清空梯度
            optimizer.zero_grad(set_to_none=True)
            if t % 2 == 0:
                print(f"in train {name} step {t} | loss {loss.item():.6f}")
            if t % num_train_steps == (num_train_steps - 1):
                checkpoint = {
                    "model": model.state_dict(),
                    "optimizer": optimizer.state_dict(),
                }
                torch.save(checkpoint, self.checkpoint_path)

    def train_loop(self):
        # 数据维度
        D = 16
        # 直接随机一个向量作为训练的目标
        true_w = torch.arange(D, dtype=torch.float32)

        # 输入数据生成方法，这样每次生成的都是不一样的输入数据，而且可以复用true_w, B就是定义的输入数据的条数
        def get_batch(B: int) -> tuple[torch.Tensor, torch.Tensor]:
            x = torch.randn(B, D)
            true_y = x @ true_w
            return x, true_y

        self.train("simple1", get_batch, D=D, num_layers=0, B=4, num_train_steps=400, lr=0.01)
        self.train("simple2", get_batch, D=D, num_layers=0, B=4, num_train_steps=10, lr=0.1)


loop_train = LoopTrain()
loop_train.train_loop()

因为模型的训练往往是不连续的，必要时必须要能恢复模型的状态，必须要知道模型的参数，以及优化器的状态，所以检查点必须要能保存和序列化这两个参数。实际的训练时还可以采用混合精度训练，一般是向前传播的中间值使用BF16或者fb8，但是参数，梯度，优化器状态使用fp32,因为这些是训练的核心必须保证精度。可以使用pytorch的AMP库或者nvidia的库来实现这两个功能。
